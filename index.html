¿Qué es la Normalización?

La normalización es un proceso de diseño de bases de datos que organiza los datos para minimizar la redundancia y evitar anomalías de inserción, actualización y eliminación.

Objetivos principales:

  • Eliminar datos redundantes
  • Mantener la integridad de los datos
  • Facilitar el mantenimiento
  • Optimizar el almacenamiento

¿Por qué es importante la normalización?

Sin normalización, las bases de datos pueden sufrir de:

  • Redundancia: Información duplicada que consume espacio innecesario
  • Anomalías de inserción: Dificultad para agregar nuevos datos
  • Anomalías de actualización: Inconsistencias al modificar datos
  • Anomalías de eliminación: Pérdida no deseada de información

Formas Normales Aplicadas

1️⃣ Primera Forma Normal (1FN)

Regla: Cada columna debe contener valores atómicos (indivisibles) y cada registro debe ser único.

❌ Tabla NO normalizada:

ID_Cliente Nombre Teléfonos Zapatos_Comprados
1 Juan Pérez 123-456, 789-012 Nike Air, Adidas Run

✅ Tabla en 1FN:

ID_Cliente Nombre Teléfono
1 Juan Pérez 123-456-7890

2️⃣ Segunda Forma Normal (2FN)

Regla: Debe estar en 1FN y todos los atributos no clave deben depender completamente de la clave primaria.

✅ Ejemplo: Tabla detalle_factura

ID_Detalle (PK) ID_Factura (FK) ID_Zapato (FK) Cantidad PrecioUnitario Subtotal
1 100 50 2 89.99 179.98
2 100 51 1 120.00 120.00

3️⃣ Tercera Forma Normal (3FN)

Regla: Debe estar en 2FN y no debe haber dependencias transitivas.

Tabla Zapato:
ID_Zapato Nombre ID_Categoria
1 Nike Air Max 1
2 Clarks Oxford 2
Tabla Categoría:
ID_Categoria Nombre
1 Deportivos
2 Formales

Beneficios del Diseño Normalizado

Eliminación de Redundancia

Los datos del cliente se almacenan una sola vez en la tabla cliente, evitando duplicación innecesaria.

Integridad Referencial

Las claves foráneas mantienen la consistencia y validez de las relaciones entre tablas.

Facilidad de Mantenimiento

Cambiar el precio de un zapato solo requiere una actualización en una ubicación.

Escalabilidad

Agregar nuevas categorías o métodos de pago es sencillo y no afecta datos existentes.

Consistencia

No hay información contradictoria almacenada en diferentes lugares de la base de datos.

Flexibilidad

Las consultas pueden combinar datos de múltiples tablas según las necesidades específicas.

Impacto en el Rendimiento:

  • Espacio de almacenamiento: Reducción significativa de datos duplicados
  • Velocidad de escritura: Menos datos para insertar y actualizar
  • Integridad: Menor riesgo de inconsistencias
  • Mantenimiento: Cambios centralizados y controlados

Creación de la Base de Datos

Crear Base de Datos y Tablas
CREATE DATABASE VentaDeZapatos; USE VentaDeZapatos; CREATE TABLE Cliente ( id_cliente INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100) NOT NULL, telefono VARCHAR(20), direccion VARCHAR(150), email VARCHAR(100) ); CREATE TABLE Empleado ( id_empleado INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100) NOT NULL, cargo VARCHAR(50) ); CREATE TABLE Categoria_Producto ( id_categoria INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(50) NOT NULL, descripcion VARCHAR(150) ); CREATE TABLE Producto ( id_producto INT PRIMARY KEY AUTO_INCREMENT, modelo VARCHAR(100) NOT NULL, talla VARCHAR(10) NOT NULL, color VARCHAR(50), precio DECIMAL(10,2) NOT NULL, id_categoria INT, FOREIGN KEY (id_categoria) REFERENCES Categoria_Producto(id_categoria) ); CREATE TABLE Metodo_Pago ( id_metodoPago INT PRIMARY KEY AUTO_INCREMENT, tipo VARCHAR(50) NOT NULL ); CREATE TABLE Factura ( id_factura INT PRIMARY KEY AUTO_INCREMENT, fecha DATE NOT NULL, id_cliente INT NOT NULL, id_empleado INT NOT NULL, id_metodoPago INT NOT NULL, total DECIMAL(10,2) NOT NULL, FOREIGN KEY (id_cliente) REFERENCES Cliente(id_cliente), FOREIGN KEY (id_empleado) REFERENCES Empleado(id_empleado), FOREIGN KEY (id_metodoPago) REFERENCES Metodo_Pago(id_metodoPago) ); CREATE TABLE Factura_Producto ( id_factura INT NOT NULL, id_producto INT NOT NULL, cantidad INT NOT NULL, precio_unitario DECIMAL(10,2) NOT NULL, PRIMARY KEY (id_factura, id_producto), FOREIGN KEY (id_factura) REFERENCES Factura(id_factura), FOREIGN KEY (id_producto) REFERENCES Producto(id_producto) ); CREATE TABLE Proveedor ( id_proveedor INT PRIMARY KEY AUTO_INCREMENT, nombre VARCHAR(100) NOT NULL, telefono VARCHAR(20), direccion VARCHAR(150), email VARCHAR(100) ); CREATE TABLE Proveedor_Producto ( id_proveedor INT NOT NULL, id_producto INT NOT NULL, PRIMARY KEY (id_proveedor, id_producto), FOREIGN KEY (id_proveedor) REFERENCES Proveedor(id_proveedor), FOREIGN KEY (id_producto) REFERENCES Producto(id_producto) );

Consultas SQL

Consultas Simples

Ver todas las facturas
SELECT * FROM Factura;
Ver proveedores con contacto
SELECT nombre, telefono, direccion FROM proveedor;
Buscar cliente específico
SELECT * FROM cliente where Nombre like "%Juan Pérez%";

Consultas Intermedias

Facturas con empleado que las gestionó
SELECT f.id_factura, f.fecha, c.nombre AS cliente, e.nombre AS empleado, f.total FROM factura f JOIN cliente c ON f.id_cliente = c.id_cliente JOIN empleado e ON f.id_empleado = e.id_empleado;
Facturas con detalle de productos
SELECT f.id_factura, f.fecha, c.nombre AS cliente, p.modelo AS producto, df.cantidad, df.precio_unitario, (df.cantidad * df.precio_unitario) AS subtotal FROM factura f JOIN cliente c ON f.id_cliente = c.id_cliente JOIN factura_producto df ON f.id_factura = df.id_factura JOIN producto p ON df.id_producto = p.id_producto ORDER BY f.id_factura;
Proveedores con sus productos
SELECT pr.nombre AS proveedor, pr.telefono, p.modelo, p.precio FROM proveedor pr JOIN proveedor_producto pp ON pr.id_proveedor = pp.id_proveedor JOIN producto p ON pp.id_producto = p.id_producto;

Consultas Avanzadas

Ingreso promedio por factura
SELECT AVG(total) AS promedio_ingresos FROM factura;
Top 3 empleados con más facturas
SELECT e.nombre AS empleado, COUNT(f.id_factura) AS total_facturas FROM empleado e JOIN factura f ON e.id_empleado = f.id_empleado GROUP BY e.nombre ORDER BY total_facturas DESC LIMIT 3;
Clientes que gastaron más que el promedio
SELECT c.nombre, SUM(f.total) AS gasto_total FROM cliente c JOIN factura f ON c.id_cliente = f.id_cliente GROUP BY c.nombre HAVING gasto_total > (SELECT AVG(total) FROM factura) ORDER BY gasto_total DESC;
Proveedor con más productos distintos
SELECT pr.nombre AS proveedor, COUNT(DISTINCT pp.id_producto) AS total_productos FROM proveedor pr JOIN proveedor_producto pp ON pr.id_proveedor = pp.id_proveedor GROUP BY pr.nombre ORDER BY total_productos DESC LIMIT 1;

Validación del Diseño

El esquema de base de datos cumple con todas las formas normales:

  • 1FN: Todos los atributos son atómicos
  • 2FN: No hay dependencias parciales
  • 3FN: No hay dependencias transitivas
  • Integridad referencial: Claves foráneas mantienen consistencia
  • Normalización BCNF: Diseño óptimo para consultas complejas
